入力タイプの仮定: **【設計・実装・事実確認】**

要望は **データ周りだけの整理**。

なので、ワークフローや実装手順は外して、**何を保存するか・どういう単位で持つか**に絞る。

---

# **データ設計の前提**

  

初期方針はこれ。

```
固定で持つものは少なくする
鮮度が必要なものは都度生成する
人間レビュー前提でシンプルにする
シリーズ（3〜5話）単位で管理する
```

つまり、

- **イベント**
    
- **状況**
    

  

は恒久マスターとして持たない。

これはAIが都度クロール・抽出・生成する。

  

固定で持つのは、

- **物語の型**
    
- **フックの型**
    

  

だけ。

  

そして、毎回の生成で使った内容と、生成後の動画情報を保存する。

---

# **保存対象の全体像**

  

初期はこの4テーブルで十分。

```
1. story_types
2. hook_patterns
3. generation_jobs
4. generated_videos
```

意味はこう。

```
story_types       = 物語型の固定マスター
hook_patterns     = フック型の固定マスター
generation_jobs   = シリーズ単位の生成条件・レビュー情報
generated_videos  = 各話単位の動画情報・レビュー情報・投稿結果
```

---

# **1. story_types**

  

## **役割**

  

物語の型を固定タグで持つ。

表記ゆれを防ぎ、分析軸にする。

  

## **持つ項目**

```
id
code
name
description
is_active
created_at
updated_at
```

## **項目の意味**

  

id

内部ID。

  

code

機械用の固定識別子。

例: misunderstanding, secret, betrayal

  

name

表示名。

例: 誤解, 秘密, 裏切り

  

description

その型の説明。

  

is_active

現在使うかどうか。

  

## **初期データ例**

```
誤解
秘密
裏切り
告白
後悔
再会
逆転
```

---

# **2. hook_patterns**

  

## **役割**

  

フック文の「文章そのもの」ではなく、**型** を持つ。

  

## **持つ項目**

```
id
code
name
template
is_active
created_at
updated_at
```

## **項目の意味**

  

code

機械用識別子。

例: found_out, thought_but, last_day

  

name

表示名。

例: 〜を知ってしまった, 〜と思ったら

  

template

テンプレート。

例: {situation}で{target}の秘密を知ってしまった

  

is_active

現在使うかどうか。

  

## **初期データ例**

```
〜を知ってしまった
〜と思ったら
最後の日に〜
〜した結果
実は〜だった
```

# **3. generation_jobs**

  

## **役割**

  

**シリーズ単位** の生成条件を持つ。

ここが中心。

  

1つの generation_job が、

1つのシリーズ企画に対応する。

  

つまり、

```
1 job = 1シリーズ
```

---

## **持つ項目**

```
id
job_uuid
trend_keywords
event_name
situation_text
story_type_id
hook_pattern_id
emotion_target
duration_sec
series_title
series_summary
episode_count
series_outline
prompt_text
prompt_review_status
prompt_review_comment
video_generation_status
final_status
created_at
updated_at
```

---

## **項目の意味**

  

job_uuid

外部参照しやすい識別子。

  

trend_keywords

元になったトレンド語。

初期は文字列で十分。

例: 卒業,新生活

  

event_name

抽出したイベント。

例: 卒業

  

situation_text

最終的な状況。

例: 卒業式の帰り道

  

story_type_id

採用した物語型。

  

hook_pattern_id

採用したフック型。

  

emotion_target

狙う感情。

例: 切ない, 共感, スカッと

  

duration_sec

各話の動画秒数。

例: 20

  

series_title

シリーズ仮タイトル。

  

series_summary

シリーズ全体の要約。

  

episode_count

話数。

例: 3, 4, 5

  

series_outline

全体シナリオ。

3〜5話全体の構成を持つ。

  

prompt_text

動画生成の元になるプロンプト全文。

初期は全体用プロンプトをここに持ってよい。

  

prompt_review_status

レビュー①の状態。

例:

- pending
    
- approved
    
- rejected
    

  

prompt_review_comment

レビュー①コメント。

  

video_generation_status

動画生成状態。

例:

- not_started
    
- generated
    
- failed
    

  

final_status

ジョブ全体の状態。

例:

- draft
    
- prompt_review_pending
    
- prompt_approved
    
- video_review_pending
    
- ready_to_post
    
- posted
    
- rejected
    

---

## **重要な点**

  

generation_jobs には、**その時使った event / situation を保存する**。

ただしそれは **恒久マスター** ではない。

  

ここを混同するとズレる。

  

正しい整理はこう。

```
Event / Situation は固定マスターにしない
でも、その回で使った内容は generation_jobs に保存する
```

# **4. generated_videos**

  

## **役割**

  

**各話単位** の動画情報を持つ。

  

つまり、

```
1 generation_job
→ 3〜5 generated_videos
```

---

## **持つ項目**

```
id
generation_job_id
episode_no
episode_title
episode_summary
episode_hook
cliffhanger_text
video_path
thumbnail_path
video_review_status
video_review_comment
post_status
platform
posted_at
views
likes
comments
shares
created_at
updated_at
```

---

## **項目の意味**

  

generation_job_id

どのシリーズから作られたか。

  

episode_no

何話目か。

例: 1, 2, 3, 4

  

episode_title

各話タイトル。

  

episode_summary

各話要約。

  

episode_hook

各話冒頭のフック文。

  

cliffhanger_text

次話への引き。

  

video_path

生成動画の保存先。

  

thumbnail_path

必要ならサムネ保存先。

  

video_review_status

レビュー②の状態。

例:

- pending
    
- approved
    
- rejected
    

  

video_review_comment

レビュー②コメント。

  

post_status

投稿状態。

例:

- not_posted
    
- posted
    
- failed
    

  

platform

投稿先。

初期は1つでもよい。

  

posted_at

投稿時刻。

  

views, likes, comments, shares

初期の最低限メトリクス。

初期は時系列で分けず、ここに直接持たせてよい。

---

# **テーブル同士の関係**

  

関係はこれだけ。

```
story_types 1 --- n generation_jobs
hook_patterns 1 --- n generation_jobs
generation_jobs 1 --- n generated_videos
```

意味としてはこう。

```
物語型とフック型を1つ選ぶ
↓
その条件で1シリーズ作る
↓
そのシリーズから3〜5話の動画ができる
```

---

# **保存しないもの**

  

初期では、あえて保存しないものも明確にする。

  

## **保存しないもの**

```
恒久 Event Master
恒久 Situation Master
投稿メトリクスの時系列スナップショット
複数候補の詳細管理
```

## **理由**

  

### **Event / Situation**

  

鮮度依存で腐る。

都度生成でよい。

  

### **投稿メトリクス時系列**

  

初期には重い。

まずは最終値だけで十分。

  

### **複数候補管理**

  

人間レビュー前提の初期では複雑すぎる。

# **初期データ設計の本質**

  

この設計でやっていることはこれ。

```
固定知識はマスターにする
その回限りの生成条件はジョブに保存する
生成された各話動画は別で持つ
投稿結果は各話に紐づける
```

つまり、

- **固定マスター**
    
- **シリーズ単位の入力**
    
- **各話単位の出力**
    

  

に分かれている。

# **この設計で後から見られること**

  

この形で保存しておけば、初期でも次は見られる。

```
どの状況がよかったか
どの物語型が強かったか
どのフック型が強かったか
何話構成がよかったか
どの話数で離脱しやすいか
```

たとえば、

- 卒業系は後悔が強い
    
- 花火大会系は告白が強い
    
- 1話目のフックが弱いシリーズは全体も弱い
    

  

みたいな分析ができる。

# **最終まとめ**

  

初期のデータ設計はこれで十分。

  

## **持つテーブル**

```
story_types
hook_patterns
generation_jobs
generated_videos
```

## **役割**

```
story_types      = 物語型マスター
hook_patterns    = フック型マスター
generation_jobs  = シリーズ単位の生成条件
generated_videos = 各話単位の動画情報と投稿結果
```

## **持たないもの**

```
恒久 Event Master
恒久 Situation Master
複雑なメトリクス時系列
```

## **基本方針**

```
イベント・状況は都度AI生成
固定で持つのは物語型とフック型
シリーズ単位で条件を保存
各話単位で動画と結果を保存
```

次に実装へ進むなら、やるべきことは **この4テーブルのDDL作成**。
